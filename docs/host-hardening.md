# 공격 표면 점검 및 호스트 하드닝 (Attacker's-view Audit & Remediation)

> "탐지만 하고 대응이 없으면 반쪽." 내 인프라를 **공격자 시점**으로 점검하고, 발견한 구멍을 우선순위대로 막은 기록. 정보보호 직무의 핵심 = *위험 식별 → 우선순위 → 조치*.

## 1. 발견 1 — 최대 피격 노드는 (예상과 달리) 미니PC였다

ES에서 호스트별로 갈라보니 직관과 반대였다.

| 서버 | 총 이벤트 | 고유 공격 IP | failure | probe | success |
|---|---|---|---|---|---|
| **mini-pc (미니PC, x86)** | **69,764** | **2,684** | 32,510 | 17,749 | 49 |
| oracle-arm (오라클, ARM) | 31,964 | 1,007 | 18,017(probe)·7,613(fail) | — | 179 |

- 미니PC가 오히려 더 두들겨 맞았다. (오라클은 6월에 로그 수집을 시작해 6월분만 존재 — 단순 비교 주의.)
- 교훈: "노출된 건 다 맞는다"가 데이터로 확인됨. 집 공인 IP로 SSH 포워딩이 열려 있던 시기의 흔적.

## 2. 발견 2 — 3월 10일, 하루 만에 70배 급감 (설정변경 시그니처)

미니PC SSH 실패를 일별로 보니:

```
02-25 ~ 03-09 : 매일 374 ~ 1,786건  (집중 폭격, 2월 피크 37,880/월)
03-09 : 1,608
03-10 :    22   ← 하루 만에 70배 절벽
04월~ : 월 2,500건 수준으로 안정
```

- 하루 만의 70배 급감은 공격자 변심으론 안 나온다 → **누가 노출을 닫은 설정변경의 시그니처.** (시기상 공개 서비스를 오라클로 이관하며 SSH 포워딩을 닫은 것으로 추정. "무엇을 했는지"는 데이터로 증명 불가 = 정황.)
- 정직 포인트: 처음엔 "4월 조치"로 기억했으나 데이터는 **3월 10일**을 가리킴. 기억보다 로그.

## 3. 발견 3 — "침해 0"은 운이 아니라 구조였다

- 오라클 sshd 실효 설정: `PasswordAuthentication no` / `PubkeyAuthentication yes` / `PermitRootLogin without-password`.
- 즉 **비밀번호 문이 없어서** 9만 번을 두들겨도 성공이 구조적으로 불가능. brute-force에 대한 *가장 강력한 조치*가 이미 적용돼 있었음.
- 성공 로그인(미니PC 49·오라클 179)은 전부 내부(도커/LAN)·known-good(가정용 회선 등) 출처. 외부 공격자의 성공 = 0건.

## 4. 조치 1 — 미니PC SSH 비밀번호 인증 차단

- 점검 결과 미니PC는 **`PasswordAuthentication yes`** (역대 최대 피격 노드인데 비번 문이 열려 있던 상태 = 진짜 구멍).
- 지금은 SSH 외부 노출이 닫혀 활성 위험은 아니나, 재노출 시 즉시 위험 → 잠재 구멍 제거.
- 조치: 드롭인 `/etc/ssh/sshd_config.d/00-hardening.conf` 에 `PasswordAuthentication no`.
- **함정 기록**: 처음 `99-hardening.conf`로 넣었으나 미적용. 원인은 sshd가 드롭인을 **알파벳순으로 읽고 먼저 읽힌 값이 이김** — `50-cloud-init.conf`(yes)가 99를 눌렀음. **`00-`로 앞 번호화**해 해결. (`sudo sshd -T`로 실효값 검증 필수.)
- 영향: 키 인증·웹 접속(내 개발용 웹 IDE)·아웃바운드(데이터 수집) 전부 무관. 순수 강화.

## 5. 발견 4 / 조치 보류 — 현재 실질 노출면은 웹(개발용 IDE)

- 미니PC의 6월 외부 트래픽은 SSH가 아니라 **웹(브라우저 기반 개발 IDE)** 탐색(해당 이벤트는 event.outcome 없음 = 인증 로그 아님).
- 구조: Caddy 리버스프록시가 여러 내부 서비스(개발용 IDE·대시보드·관리 콘솔 등)를 서브도메인으로 노출. DNS는 Cloudflare 없이 **집 공인 IP 직노출(IP 비공개)**, 80/443 포워딩.
- 그 개발용 IDE는 **터미널·셸 접근이 가능** = 뚫리면 사실상 서버 장악(RCE). 자체 로그인은 있으나 **앞단에 추가 인증이 없어** 단일 비밀번호에 의존.
- 권장: Caddy `basic_auth` 2차 방어(어디서든 접속 유지하며 자물쇠 추가) 또는 Cloudflare Access/Tunnel(집 IP 은닉). **현재 보류**(위생 목적, 우선순위 낮음).

## 6. 한 줄 요약

내 서버를 공격자 시점으로 점검 → 데이터로 **미니PC가 최대 피격·SSH 비번 문 열림**을 발견 → 비번 인증 차단(키 전용) → 실질 노출면이 SSH에서 **웹(셸 가능 IDE)** 으로 이동했음을 식별하고 2차 방어를 설계. *원천 차단이 탐지보다 우선*이라는 관점의 실천.
