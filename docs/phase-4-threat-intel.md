# Phase 4 (v2) — 외부 위협 인텔 연동 (AbuseIPDB)

> v1(Phase 1~3 + 하드닝)은 **내 서버 로그라는 내부 신호만으로** 공격을 판정했다. 한계는 "내가 본 것만 안다"는 것 — 처음 보는 IP와 전 세계 상습범을 구분하지 못한다. v2는 외부 위협 인텔(AbuseIPDB)을 연동해 **"내 판정이 외부 세계와 일치하는가"**를 검증하고, 평판을 탐지·알림에 반영해 더 견고하게 만든 단계.

## 1. 동기 — v1의 한계

- v1의 탐지는 빈도(5분 20회)·성공여부·계정 같은 **내부 신호**만 본다.
- 그래서 "이 공격자가 우리만 노린 신규 IP인가, 아니면 전 세계가 이미 아는 상습범인가"를 모른다.
- 보안 성숙도에서 외부 위협 인텔 연동은 전형적인 다음 단계. "정보보호 데이터 엔지니어 = 외부 피드를 수집·정규화·조인하는 파이프라인"이라는 직무 정의와도 정확히 맞는다.

## 2. 검증 분석 — 우리 공격자의 평판

상위 공격자 200개(실패 횟수 기준, known-good·사설 제외)를 AbuseIPDB `/check`로 조회.

| 평판(abuseConfidenceScore) | 비율 |
|---|---|
| 100% (확정 상습범) | 88개 (44%) |
| ≥75% | 91개 (45%) |
| ≥25% (신고 이력 있음) | **100개 (50%)** |
| 0% (어떤 신고도 없음) | 54개 (27%) |

- 평균 점수 49%, **평균 신고 7,042건**, 누적 신고 **140만 건**.
- 해석 1: **우리 공격자의 절반이 이미 전 세계에 신고된 상습범** → 우리 탐지가 실제 위협을 잡고 있다는 외부 검증.
- 해석 2: **27%는 어떤 피드에도 없다** → 평판만 믿으면 1/4을 놓친다. 외부 인텔은 *보강*이지 *대체*가 아니다. (이 균형 인식이 핵심 인사이트.)

## 3. 아키텍처 — 2계층 설계

```
[일일 갱신]  AbuseIPDB /blacklist (confidence>=90, 1만 IP)
              → threat-intel-abuseipdb 인덱스 재적재 → enrich 정책 재실행
              (systemd timer, 매일 05:00 UTC)

[벌크 enrich] 수집 파이프라인(seclogs-parse)에 enrich 프로세서 삽입
              → 들어오는 이벤트의 source.ip를 블랙리스트와 매칭 → threat_intel.* 태깅

[알림시 조회] 포워더가 탐지된 IP를 AbuseIPDB /check로 실시간 조회(1건/알림)
              → Slack 카드에 "점수 X% / 신고 Y건 / 상습범" 표시
```

**왜 2계층인가 (정직한 트레이드오프):**
- `/blacklist`는 **전 세계 top-10k**라 우리 특정 공격자를 다 덮지 못한다. 실제로 우리 주 공격자 `193.163.187.x`은 개별 조회 시 100%인데 블랙리스트(top-10k)엔 없었다.
- 그래서 **벌크 enrich(근사, 무료·저비용, 상시)** + **알림 시 per-IP check(정밀, 탐지 건수만큼만 호출 → 무료 한도 내)**로 분리. 비용·커버리지·정밀도의 균형.

## 4. 구현 디테일

- enrich 소스: `threat-intel-abuseipdb` 인덱스(ip keyword, abuse_score, last_reported, country). enrich 정책 `abuseipdb-policy`(match on ip).
- ingest: `seclogs-parse`에 `enrich` 프로세서 추가(`field: source.ip`, `target_field: threat_intel`, `if: ctx.source?.ip != null`). simulate로 검증 후 PUT.
- 포워더: `ti_check(ip)`가 `/check` 호출(타임아웃·실패 시 None → 알림은 계속). 평판은 **정보 표시용 컨텍스트**이고, 심각도 로직(영향 기반: 성공→critical 등)은 그대로. "known-bad + 성공"은 이미 성공으로 critical.
- 시크릿: API 키는 `/etc/threat-intel.env`(600)에만. 코드·git 미포함. 포워더 서비스는 `EnvironmentFile`로 주입.
- 갱신: `threat-intel-refresh.timer`(매일) → 피드 신선도 유지.

## 5. 결과와 의의

- **검증**: "우리 공격자의 50%가 글로벌 상습범, 누적 신고 140만 건" — 내부 탐지를 외부로 교차검증.
- **인사이트**: "27%는 미신고 → 평판은 보강이지 대체가 아니다"는 균형 잡힌 판단.
- **데이터 엔지니어링**: 외부 피드 수집 → ES enrich 정책 → ingest 조인 → 일일 갱신 자동화. 정확히 직무가 하는 일.
- **성장 서사**: v1을 완성하고 한계를 인식해 v2로 발전. "내가 본 것만 안다 → 외부 세계와 대조한다"로의 도약.

## 6. 한계 / 다음

- 블랙리스트 커버리지(top-10k)는 우리 공격자를 부분만 덮음 → 알림 시 per-IP로 보완하나, 전수 enrich는 아님.
- 단일 피드(AbuseIPDB). Spamhaus·GreyNoise 등 다피드 교차 시 신뢰도 향상 여지.
- 평판을 심각도에 *직접* 반영하진 않음(현재 정보 표시). known-bad 가중치 룰은 다음 과제.
